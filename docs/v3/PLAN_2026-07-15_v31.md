# AVA v3.1 Direction — Post-C5 Plan (2026-07-15)

> Written after the C5 SFT run + a discussion/consultation phase. C5 (plain SFT
> on OpenCodeReasoning + CommitPackFT) did its job — **held donor-level code
> while gaining reasoning** (probe @2167: HumanEval+ 80 vs donor 85 = flat in
> noise; MBPP+ 68.33 = exactly flat; MMLU +13). Per our own stop rule, SFT on
> these benches is **saturated**: the donor is near its 4-bit pass@1 ceiling and
> more SFT epochs won't move HumanEval+/MBPP+. The gains now live elsewhere.
> This doc records what "elsewhere" is and the decisions from consultation.

## 1. The strategic frame (validated, not new)

The user's thesis — *"an SLM wins on performance/optimization, not scale"* — is
`EDGES.md`'s founding argument. Post-SFT, value comes from **orthogonal cheap
axes**, not more parameters or more epochs. Concretely, three levers:

- **Harder + agentic data** (measurable ceiling the donor hasn't maxed).
- **Test-time compute** (internal loops / recurrent depth — buy accuracy with
  inference seconds, no new params).
- **Execution-verified self-play** (the only path that *beats* the donor rather
  than matching it — grounded in correctness, not imitation).

## 2. The legal line (binding)

**No proprietary-model distillation.** Claude-Fable-5 / Claude-Mythos-5 /
GPT-5.5-Codex trace datasets are ToS-violating to train a released competitor,
and they poison AVA's entire identity (`$0, reproducible, free forever, cleanly
releasable`). The uploader's license can't grant rights to outputs they don't
own. Dropped from the mix: `Glint-Research/Fable-5-traces`,
`armand0e/claude-fable-5-claude-code`, `WithinUsAI/claude_mythos_distilled_25k`,
`Crownelius/Complete-FABLE.5-traces-2M`, `HelioAI/Claude-Fable-5-5500x`,
`AletheiaResearch/GPT-5.5-Codex`, and the analogous distilled *models* as data.

**The clean substitute is a higher ceiling, not a compromise** (consultation):
open-teacher distillation + self-play + structural transfer beats proprietary
distillation for a 4B execution-verified student, because (a) the 4B is
capacity-bottlenecked so the teacher-quality gap shrinks, (b) self-play grounds
on real correctness rather than a teacher's approximation, (c) reasoning
*structure* (plan→code→verify→refine) is learnable on clean data.

## 4. Method plan

- **DoRA / rsLoRA** — drop-in upgrade over vanilla LoRA r=16 (better gradient
  behavior, same param/compute budget). Cheap, immediate. *(shipping now)*
- **Recurrent-depth latent reasoning** — Geiping et al. arXiv:2502.05171
  ("Huginn"): a 3.5B reaches ~50B-equivalent compute by looping a recurrent
  block at test-time, **no special training data**, dial-able depth. This is
  the HRM refinement loop's published cousin — our `engine/hrm_core.py` already
  encodes the idea; adopt the cleaner recipe. *(needs compute; C-loop phase)*
- **Self-play / execution RL** (C6, Absolute Zero) — the differentiator. Propose
  → solve@high-budget → verify in sandbox → distill verified@low-budget.
  *(needs compute)*
- **Structural transfer** — encode plan→code→self-check→refine as the trained
  format contract, learned on clean/self-generated data.
- **AIDE autoresearch** (WecoAI, MIT, runs local) — two uses: (a) auto-tune our
  own data mix / LR / rank to develop v3 faster; (b) the concrete engine for the
  S3/S4 self-improvement rungs. *(evaluate; tool + capability)*

## 5. Evaluation upgrade (the missing instrument)

HumanEval+/MBPP+ are saturated vs the donor. Add **harder evals** to measure the
specialization that actually matters:
- **SWE-bench Lite / Verified** (princeton-nlp) — agentic, via the harness.
- **MiniMaxAI/OctoCodingBench** — broader coding eval.
- **CanItEdit / edit-task probe** on CommitPack — measure the hashline edit skill.

Strict decontam: all eval sets stay out of the training mix (existing R21 gate).

## 6. Inference/compression note

**We are already past every general-purpose weight-compression method on the
bits/weight axis — they solve a problem we don't have.** Surveyed and parked:

| Method | bits/weight | Built for |
|---|---|---|
| Lossless BF16 exponent coding (ZipNN/DFloat11/brianbell) | ~11 | datacenter storage of huge FP weights, bit-exact |
| NVFP4 (near-BF16, lossy) | ~4 | Blackwell/vLLM serving, 32 GB class |
| **AVA ternary TQ1_0/TQ2_0 (our plan)** | **~1.7** | **4 GB laptop, llama.cpp** |

All these shrink *big→medium at high quality*; AVA shrinks *tiny→fits-4GB* by
deliberately discarding precision (lossy ternary QAT). Lossless 11-bpw / FP4
4-bpw don't fit 4 GB and don't run on llama.cpp. Weight compression is a **solved
and maxed axis for us** — gains come from the *other* axes (MTP speculative
decode, test-time loops, self-play, better data), never more weight compression.
Revisit only if serving a *big teacher* locally where lossless storage helps.
