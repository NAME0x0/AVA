> Scope: read the full `docs/v3/` set + `experiments/exp6_v3/` engine/scripts against
> two questions — (1) is anything broken or contradictory, (2) how does v3 become a

## 1. The one that blocks everything: the 4 GB budget was never redone for the donor

`ARCHITECTURE_V3.md` §2 says the packed model is **1.70 GB** and fits 4 GB with headroom.
That table is for the dead **clean-slate 24×1792 / vocab-248320** model. The actual model
is a **Qwen3-4B donor** (hidden **2560**, **36** layers, FFN **9728**, vocab 151936).
Nobody recomputed the fit. Doing it (numbers reproducible, `scripts` calc in the review commit):

| Path | Routed experts | Total weights | Fits 4 GB (weights+KV+act)? |
|---|---|---|---|
| **MoE as specced** (32×1216 experts + full-donor shared, ternary) | 10.76 B → 2.27 GB | **≈ 4.0 GB** | **NO** — over before KV/activations |
| **Dense-ternary donor** (drop MoE; keep Mamba hybrid + HRM) | — | **≈ 1.73 GB** | **YES**, ~3 GB resident |
| Bare donor at Q4 (floor product) | — | ≈ 2.0 GB | YES |

**Why**: MoE-ifying a dense FFN *multiplies* the FFN parameter class (32×1216 ≈ 4× the dense
9728), and ternary only buys ~4.7× over BF16 — the two nearly cancel, then the shared
full-donor copy is added on top. The MoE is simultaneously (a) the component that breaks the
4 GB target and (b) the riskiest stage (R4: no published 1.58-bit MoE > 7 B). Their own R4
"Fallback C" is *"drop MoE, ship dense ternary."* **That should be the default, not the fallback.**

Action: rewrite the `ARCHITECTURE_V3.md` param/memory table for the donor, and re-tier the MoE
from "core" to "gated ablation that must beat the dense-ternary baseline AND stay < 3.6 GB resident to ship."

## 4. Feasibility, honestly

**As currently scoped (full exotic stack): not realistic for one person on free compute in 3–6 months.**
The critical path stacks five novel, interacting, failure-prone stages — LoLCATs linearization of 27
layers → sparse-upcycle to 32 ternary experts → BitDistiller QAT recovery → HRM latent curriculum →
execution-RL self-play — each needing 1–2 debug/redo cycles, each on **preemptible 3-hour shards**.
Any one stalling (R2 Mamba BF16 instability, R4 ternary-MoE cliff, R1 HRM non-transfer) cascades.
The docs are unusually honest about each risk individually; what's missing is that the **joint**
probability of all five landing, solo, on T4 quota, is low.

**Staged scope: very realistic, and still a huge v2 jump.** See §5.

## 5. How v3 actually becomes a big jump — stage the ambition

The largest v2→v3 gain is *free the moment you warm-start from the donor*. v2 = 2 B QLoRA,
HumanEval+ 19.5 %, MMLU 59. Qwen3-4B alone is dramatically higher on every axis. So:

**v3.0 (ship this — the safe product that already crushes v2 and is competitive in-class):**
- Donor Qwen3-4B, **kept dense** (no MoE), ternary/Q4 QAT for the 4 GB fit (§1 shows it fits at ~1.7 GB).
- Coding SFT (OpenCodeReasoning + CommitPackFT + FIM) + multi-teacher **trace** distillation.
- **Test-time compute is the equalizer**: HRM refinement loop *or* plain self-consistency/best-of-n
  against the free verifier. This is the single lever that lets a 4 GB model punch at 7–14 B on
  agentic/SWE tasks — the same trick frontier tiny models use.
- **MCP + sandbox harness shipped as the product** (model + executor as one unit).
- Self-play flywheel (S1–S2) for continual gains on idle hours.
- Mamba-3 hybrid is **optional here** — only needed for the long-context/throughput headline, and it
  carries R2/R22. Ship v3.0 full-attention if Mamba wobbles.

### 5b. Quantifying "significant jump" — the jump is guaranteed by inheritance, and that changes what the bar is

v2 = Qwen3.5-2B QLoRA. The v3 donor (Qwen3-4B) beats every v2 headline number **out of the box,
before a single v3 mechanism runs** (donor figures ≈ published Qwen3-4B class; exact values land at C1
— that is what C1 is for):

| Benchmark | v2 (measured) | v3 target (gap doc) | **Unmodified donor (≈)** |
|---|---:|---:|---:|
| HumanEval+ | 19.5 | ≥ 50 | **~70–80** |
| MBPP+ | 35.7 | ≥ 50 | **~65–75** |
| GSM8K greedy | 35.3 | ≥ 70 | **~80–90** |
| MATH-500 | 18.8 | ≥ 50 | **~60+ (thinking)** |
| MMLU | 59.2 | ≥ 70 | **~70** |
| IFEval strict | 31.6 | ≥ 65 | **~80** |

Consequences:

1. **"Significant jump over v2" is a solved problem** — adopted the day the donor is warm-started.
   Nothing exotic is needed to clear it. Ship donor + QAT-4GB fit + harness and v2 is dominated by
   30–60 pp on the axes that matter.
2. **The real bar is therefore the donor, not v2.** Every transplant stage (T1 linearize, T2 MoE-ify,
   QAT) *risks degrading below donor* — the exotic stack must earn back what surgery costs before it
   adds anything. CODE_PIVOT §7 already encodes this ("beating our own donor decisively —
   non-negotiable"); the per-stage ≤ 2 pp regression gates are the right instrument. The framing just
   needs to flip everywhere else: v3's success metric is **delta over donor**, not delta over v2.
3. Where v3 is a *genuine* product jump beyond "rebadged Qwen at Q4": (a) the execution harness +
   agentic layer (v2 tool-call rate 0.6 % → a real MCP loop), (b) test-time compute scaling
   (refinement/self-consistency vs the free verifier), (c) self-play continual improvement on idle
   hours, (d) coding specialization via multi-teacher traces — a 4 B specialist can beat 7–14 B
   generalists in-domain. Those four are the v3 story. HRM/MoE/Mamba/PKM are optimizations on top,
   each gated on beating the v3.0 baseline.

**v3.1+ (the research bets, each must beat the v3.0 baseline to merge — not shipped by default):**
- Ternary **MoE** upcycle (only if it beats dense-ternary at equal VRAM *and* stays < 3.6 GB).
- **Mamba-3** conversion (only if RULER + the multi-hop probe R22 hold).
- **PKM RAM-tier memory** (E1, gated by A8), **latent curriculum** (E2, A9), **NSA** (E4).
- **S3–S4** self-improving pipeline — headline-grade if it lands, rabbit-hole if it's on the critical path.

This inverts the current default (exotic = core, safe = fallback) into (safe = product, exotic =
gated experiment). It matches the docs' own "any phase can ship" ladder — just makes the shippable
rung the *first* one, so there is always a releasable artifact.

## 6. Frontier-tiny reality check (set expectations correctly)

gpt-oss-20B, Haiku, Flash are 20 B+ active or proprietary — **you will not match them on general
capability at ≤ 4 GB.** What is achievable and defensible:
- **Best open coder in the ≤ 4 GB class** (beat the donor decisively; beat every ≤ 4 B open coder).
- **Competitive with 7–14 B on *agentic* coding** via the harness + test-time compute + self-repair.
- The honest headline is *"$0, 4 GB, laptop-trained, and it out-codes its weight class while
  harassing models 3× its size on SWE-style tasks"* — which the CODE_PIVOT tier table already states
  correctly. Keep that framing; drop the "strictly dominate every benchmark" language from INDEX.

## 7. Checkpoint / artifact strategy across Colab + Kaggle (your explicit ask)

Foundation already exists and is good: `scripts/checkpoint_sync.py` — HF Hub as single source of
truth, atomic `LATEST.json`-last upload, RNG + step capture, resume-from-anywhere. Fixes applied in
this review:
- **`trainable_only=True`** — pushes only `requires_grad` params (LoRA adapters + router + norms +
  mamba/halting projections), not the frozen multi-GB donor base. Turns each 30-min checkpoint from
  GB → MB, which is the difference between fitting free quota and not. Resume uses `strict=False`
  (base loaded from the donor repo first, adapters layered on).
- Test added (`test_trainable_only_saves_subset_and_resumes`), full suite green (4/4).

Operating model:
1. **HF Hub private repo** `NAME0x0/AVA-v3-checkpoints` = canonical state. `HF_TOKEN` via env/secret only.
2. Each session **time-boxes** to under the platform limit (Kaggle ~160 min, Colab ~150 min),
   `maybe_save` every 30 min, force-save on exit. Preemption costs ≤ 30 min by construction.
3. **Deterministic data order** (seed + step) → on resume, fast-forward the stream to the resumed
   step so resumed data = never-seen data (implemented in the notebook trainer).
4. **Rotate platforms freely**: Kaggle out of quota → Colab → laptop overnight, same repo, same run.
5. Phase gate: soup last-k checkpoints (`checkpoint_ema_last_k: 4` already in the donor YAML), export
   the winner to `NAME0x0/AVA-v3` (BF16), GGUF later.
6. Keep datasets/teacher-traces in their own Hub dataset repos (R17 hardware-loss mirror).

Gap to close later: the trainer must also snapshot the **streaming dataset shard id** for datasets
whose order isn't purely seed-derived; the fast-forward assumes a deterministic generator.

## 8. Can you train on the RTX A2000 4 GB? (your explicit ask)

**Partially — treat the laptop as the always-on coordinator + inference box, Kaggle as the training workhorse.**

| Workload | Laptop 4 GB | Where it should run |
|---|---|---|
| Donor QLoRA (4-bit base + LoRA + paged 8-bit optim + grad-ckpt + seq ≤ 1024, mb=1) | marginal, slow (~1 K tok/s) | small ablations laptop; real shards **Kaggle/Colab** |
| T1 LoLCATs (needs donor attn outputs as targets — precomputable) | tight | **Kaggle** |
| T2 sparse-upcycle + QAT recovery | no (needs donor + student resident) | **Kaggle/Colab** |
| Teacher trace generation (streaming int4, overnight) | **yes** (Exp5 loader, slow but free) | **laptop** |
| Self-play inference + sandbox verify | **yes** | **laptop** |
| Eval matrix (needs full donor at speed) | mostly no at 4 GB | **Kaggle** |
| μP tiny-proxy sweeps (50–200 M) | **yes** | **laptop** |
| Orchestration / checkpoint sync / data prep | **yes** | **laptop** |

32 GB system RAM is the real asset — it's what makes CPU-offload optim and the E1 RAM-tier feasible.
The 4 GB VRAM is a *deployment* target the donor already meets at Q4 (§1); it is **not** enough to be
the primary trainer. That's exactly why the notebook is Colab/Kaggle-first with laptop resume.

## 11. Harness selection (2026-07-02)

v3 needs harnesses in **three distinct roles** — eval, product, trace-gen — and one harness cannot
fill all three. Field surveyed: pi, oh-my-pi (OMP), Hermes, Factory droid, opencode, goose,
OpenHands, aider, mini-swe-agent.

| Harness | License | Prompt size | Local/OpenAI-compat | Small-model fit | Verdict |
|---|---|---|---|---|---|
| **mini-swe-agent** (SWE-bench team) | MIT | minimal, **bash-only, no tool schema** | any endpoint | excellent — burden on model, not scaffold | **ADOPT: eval + SWE-trace harness** |
| **pi** (badlogic/Zechner) | MIT | **< 1,000 tokens, 4 tools** | Ollama/llama.cpp/15+ providers | best-in-field — only harness sized for a 4B | **ADOPT: product base — fork as the AVA harness, wire our FastMCP sandbox** |
| oh-my-pi (OMP, can1357 fork of pi) | MIT | 32 tools — schema bloat for 4B | 40+ providers incl. llama.cpp | mixed | **don't adopt wholesale; STEAL hashline** (hash-anchored edits: −61 % output tokens, ~2× pass rate for small models) + LSP-tool idea |
| Hermes (Nous Research, Feb 2026) | MIT | heavy (gateway, chat platforms, memory) | any endpoint, local auto-detect | moderate | watch — skill-persistence pattern = philosophical cousin of S3/S4; revisit post-v3.0 |
| goose (Block → Linux Foundation) | Apache 2.0 | heavy, MCP-native 70+ ext | Ollama + 15 providers | moderate | optional demo consumer of v3's MCP server |
| opencode | MIT | mid | 75+ providers | good | compatibility target, not dev harness — v3 via llama-server is usable from it for free |
| Factory droid | proprietary, cloud coordinator | n/a | BYOK but cloud-hosted logic | n/a | **OUT — violates D3 air-gap, $0, free-forever** |
| OpenHands | MIT | heavy, container env | yes | poor at 4 GB | out (footprint) |
| aider | Apache 2.0 | mid, diff-based | 100+ via LiteLLM | good | eval only (Aider polyglot already in matrix) |

**Decisions:**

1. **Eval = mini-swe-agent, non-negotiable.** SWE-bench team's own harness, ~100 lines, >74 %
   Verified with frontier models. Bash-only design measures the *model*, not the scaffold — and the
   2026 literature ("Stop Comparing LLM Agents Without Disclosing the Harness", arXiv 2605.23950)
   makes harness disclosure mandatory for publishable numbers. v3's SWE-bench claims run on it.
2. **Product = AVA harness, forked from pi.** Sub-1K-token system prompt is the only one a 4B can
   afford (Claude-Code-class prompts eat 10 %+ of useful context and drown small models). MIT,
   built-to-fork, session trees = inspectable JSON → direct D9 glass-box synergy. Our FastMCP
   sandbox mounts as pi extensions; XGrammar constrained decoding stays ours.
3. **Steal OMP's hashline edit format as a training dialect.** Hash-anchored edits doubled
   small-model pass rates and cut output tokens 61 % — train v3 to emit hashline natively (C5 edit
   tier alongside CommitPackFT diffs; feeds D5 diff-minimality). Data-format adoption, not harness adoption.
4. **Harness–model co-design, two dialects only:** (a) bash-only mini-swe-agent dialect (eval +
   SWE traces), (b) AVA-pi MCP dialect (product). A 4B cannot learn every harness's conventions;
   teacher-panel traces are rendered into exactly these two.
5. Leads logged for C6: SWE-Universe (arXiv 2602.02361, millions of verifiable envs) as self-play
   environment source; Polar (arXiv 2605.24220, agentic RL on any harness) for the RL lane.
